# Auditoría

# 1. Activa desde SQL*Plus la auditoría de los intentos de acceso exitosos al sistema. Comprueba su funcionamiento.

Visualizaremos los parametros de auditorias:

show parameter AUDIT

Nos aseguraremos de que el parámetro AUDIT_TRAIL esté establecido en DB. Puedes verificarlo ejecutando el siguiente comando:

show parameter audit_trail

Si el valor devuelto no es DB, puedes establecerlo con el siguiente comando:

alter system set audit_trail=db scope=spfile;

Activamos la auditoría para los intentos de acceso exitosos con el siguiente comando:

audit session whenever successful;

Comprobamos su activación.

audit create session by lucas;

select * from dba_priv_audit_opts;

Comprobamos el funcionamiento:

Primero iniciaremos sesión en la base de datos SQL*Plus con una cuenta de usuario existente con cualquiera de los siguientes comandos:

sqlplus

sqlplus username/passwd

Si la autenticación es exitosa, la sesión se iniciará y deberías ver el prompt de SQL*Plus. Puedes comprobar que la auditoría está funcionando consultando la tabla de registros de auditoría dba_audit_session. Puedes hacerlo ejecutando el siguiente comando:

select os_username, username, extended_timestamp from dba_audit_session where action_name = 'LOGON' and returncode = 0;

Para finalizar, desactivaremos la auditoría.

noaudit session whenever successful;

# 2. Realiza un procedimiento en PL/SQL que te muestre los accesos fallidos junto con el motivo de los mismos, transformando el código de error almacenado en un mensaje de texto comprensible. Contempla todos los motivos posibles para que un acceso sea fallido.

En primer lugar debemos de activar la auditoria de los aciertos fallidos:

audit create session whenever not successful;

Función para devolver el motivo del error:

create or replace function DevolverMotivo (p_error number)
return varchar2
is
    mensaje varchar2(40);
begin
    case p_error
        when 1017 then 
            mensaje:='Usuario/Password Incorrecta';
        when 28000 THEN
            mensaje:='Cuenta Bloqueada';
        when 1045 then
            mensaje:='Privilegios insuficientes';
        else
            mensaje:='Error Desconocido';
    end case;
return mensaje;
end;
/

Procedimiento:

create or replace procedure MostrarAccesosFallidos
is
    cursor c_cursor is 
    select username, timestamp, returncode from dba_audit_session where action_name = 'LOGON' and returncode != 0;
    v_motivo varchar2(40);
begin
    dbms_output.put_line(chr(10)||chr(9)||chr(9)||'-- AUDITORIA DE ACCESOS FALLIDOS --');
    dbms_output.put_line(chr(10)||chr(9)||'Usuario'||chr(9)||chr(9)||'Fecha'||chr(9)||chr(9)||chr(9)||
        'Motivo');
    dbms_output.put_line(chr(9)||'----------------------------------------------------------------');
    for var in c_cursor loop
        v_motivo:=DevolverMotivo(var.returncode);
        dbms_output.put_line(chr(10)||chr(9)||var.username||chr(9)||chr(9)||to_char(var.timestamp,'YY/MM/DD DY HH24:MI')||chr(9)||v_motivo);
    end loop; 
end;
/

Comprobación del funcionamiento:

Para hacer la prueba voy a acceder con los usuarios "lucas", "oscar" y "oscarlucas". Con el usuario "lucas" voy a fallar en la contraseña, el usuario "oscar" esta bloqueado y el usuario "oscarlucas" no tiene privilegios.

Y ejecutaremos el procedimiento para ver su funcionamiento:

exec MostrarAccesosFallidos;

Para finalizar, desactivamos la auditoria

noaudit create session whenever not successful;

# 3. Activa la auditoría de las operaciones DML realizadas por SCOTT. Comprueba su funcionamiento.

Esquema scott (opens new window)

Activamos la auditoría.

audit insert table, update table, delete table by scott by access;

Donde:

  • BY ACCESS : Realiza un registro por cada acción.

  • BY SESSION : Realiza un registro de todas las acciones por cada sesión iniciada.

Realizamos una prueba:

conn scott/tiger
insert into dept values(50, 'DH', 'Dos Hermanas');
update dept set loc = 'Sevilla' where deptno = 50;
delete from dept where deptno = 50;
commit;

Para ver las acciones realizadas por SCOTT:

audit all by scott by access;

select obj_name, action_name, timestamp from dba_audit_object where username = 'SCOTT';

# 4. Realiza una auditoría de grano fino para almacenar información sobre la inserción de empleados con sueldo superior a 2000 en la tabla emp de scott.

Creación de la auditoría de grano fino:

begin
    dbms_fga.add_policy (
        object_schema      =>  'scott',
        object_name        =>  'emp',
        policy_name        =>  'politica1',
        audit_condition    =>  'sal > 2000',
        statement_types    =>  'insert'
    );
end;
/

Ver la política creada:

select object_schema,object_name,policy_name,policy_text from dba_audit_policies;

Realizamos una prueba:

conn scott/tiger

insert into emp values(7940, 'MARK', 'CARRIER', null, sysdate, 3000, null, 10);
insert into emp values(7941, 'RICK', 'CARRIER', null, sysdate, 2001, null, 20);

commit;

Para visualizar las acciones realizadas.

select sql_text from dba_fga_audit_trail where policy_name = 'POLITICA1';

Para eliminar la auditoría de grano fino:

begin
    dbms_fga.drop_policy (
        object_schema      =>  'scott',
        object_name        =>  'emp',
        policy_name        =>  'politica1'      
    );
end;
/

# 5. Explica la diferencia entre auditar una operación by access o by session ilustrándolo con ejemplos.

BY ACCESS:

La auditoría por acceso implica registrar una entrada de auditoría cada vez que se lleva a cabo una operación de manipulación de datos (DML). Por ejemplo, esta técnica de auditoría permite registrar si un usuario ha insertado, actualizado o eliminado una fila de una tabla.

Primero vaciamos la tabla de auditoria eliminando los registros

truncate table aud$;

Despues activamos la auditoría.

audit insert table, update table, delete table by scott by access;

Realizaremos una serie de pruebas.

conn scott/tiger

insert into scott.dept values(60, 'OLL', 'ESPAÑA');
update scott.dept set loc = 'MIAMI' where deptno = 60;
delete from scott.dept where deptno = 60;

commit;

Cambiamos al usuario administrador y comparamos registros:

select username, action_name, timestamp, obj_name from dba_audit_object where username = 'SCOTT';

BY SESSION:

La auditoría por sesión, en contraste con la auditoría por acceso, registra únicamente una entrada de auditoría por sesión. Aunque esta técnica de auditoría proporciona un registro menos detallado, mejora el rendimiento de la base de datos.

Primero vaciamos la tabla de auditoria eliminando los registros

truncate table aud$;

Despues activamos la auditoría.

audit insert table, update table, delete table by scott by session;

Realizaremos una serie de pruebas.

conn scott/tiger

insert into scott.dept values(60, 'OLL', 'ESPAÑA');
update scott.dept set loc = 'MIAMI' where deptno = 60;
delete from scott.dept where deptno = 60;

commit;

Cambiamos al usuario administrador y comparamos registros:

select username, action_name, timestamp, obj_name from dba_audit_object where username = 'SCOTT';

Las diferencias son mínimas y es que solo se diferencia en que db extend aparte de almacenarse en SYS.AUD$ también escribe valores en las columnas SQLBIND y SQLTEXT.

# 7. Averigua si en Postgres se pueden realizar los cuatro primeros apartados. Si es así, documenta el proceso adecuadamente.

A diferencia de otras bases de datos, PostgreSQL no incluye de forma nativa una herramienta para realizar auditorías. Sin embargo, la comunidad ha desarrollado una herramienta llamada Audit trigger 91plus para solventar esta limitación.

Audit trigger 91plus es una herramienta creada por la comunidad de PostgreSQL que nos permite auditar las operaciones realizadas en la base de datos, tanto las de manipulación de datos (DML) como las de definición de objetos (DDL).

Para instalar Audit trigger 91plus, es necesario seguir algunos pasos sencillos, como descargar el archivo de instalación, crear una base de datos de auditoría y ejecutar el archivo de instalación en la base de datos principal. Una vez instalado, Audit trigger 91plus registrará las operaciones realizadas en la base de datos y las almacenará en la base de datos de auditoría creada previamente.

Gracias a Audit trigger 91plus, se puede realizar un seguimiento exhaustivo de todas las operaciones realizadas en la base de datos, lo que aumenta la seguridad y el cumplimiento normativo de la misma.

Descargamos la herramienta:

wget https://raw.githubusercontent.com/2ndQuadrant/audit-trigger/master/audit.sql

Una vez descargada, la importamos en la base de datos:

\i audit.sql

Una vez que hemos habilitado la auditoría con Audit trigger 91plus, si queremos ver las operaciones DML realizadas por un usuario concreto, como puede ser SCOTT, tendremos que especificar las tablas que queremos auditar, ya que no es posible realizar una auditoría global de todas las tablas de la base de datos:

SELECT audit.audit_table('emp');
SELECT audit.audit_table('dept');

Para pobrar que la auditoría funciona, vamos a hacer cambios en las tablas:

insert into dept values (50, 'NULO', 'MIAMI');
insert into emp values (9000, 'KING','PRESIDENT', null, to_date('25-02-2023','dd-mm-yyyy'), 5000, null, 10);

update dept set loc = 'DH' where deptno = 50;
update emp set ename = 'QUEEN' where empno = 9000;

delete from dept where deptno = 50;
delete from emp where empno = 9000;

Ahora, vamos a comprobar que la auditoría ha funcionado correctamente:

select session_user_name, action, table_name, action_tstamp_clk, client_query from audit.logged_actions;

# 8. Averigua si en MySQL se pueden realizar los apartados 1, 3 y 4. Si es así, documenta el proceso adecuadamente.

Primero habilitamos las auditorias, debemos instalar un plugin adicional de mariadb:

sudo mysql

install soname 'server_audit';

Una vez hecho esto, para habilitar la auditoría DML de scott, debemos modificar la configuración de mariadb /etc/mysql/mariadb.conf.d/50-server.cnf con lo siguiente para solo coger las conexiones:

[server]
server_audit_events = CONNECT
server_audit_logging = ON

Ahora reiniciamos el servicio de mariadb para aplicar los cambios:

sudo systemctl restart mariadb

Ahora, vamos a comprobar que la auditoría ha funcionado correctamente mirándolo en los logs de MySQL:

Primero nos conectaremos con un usuario

mysql -u lucas -p

Y ahora miraremos los logs

sudo cat /var/lib/mysql/server_audit.log

Hbilitamos la auditoría DML de scott, debemos modificar la configuración de mariadb (/etc/mysql/mariadb.conf.d/50-server.cnf) con lo siguiente:

[server]
server_audit_events=CONNECT,QUERY,TABLE
server_audit_logging=ON
server_audit_incl_users=scott

Ahora reiniciamos el servicio de mariadb para aplicar los cambios:

systemctl restart mariadb

Ahora, entramos con el usuario scott y ejecutamos algunas sentencias, para ver si se registran adecuadamente:

insert into scott.dept values(60, 'OLL', 'ESPAÑA');
update scott.dept set loc = 'MIAMI' where deptno = 60;
delete from scott.dept where deptno = 60;

Miremos en el fichero de auditorías para ver si han guardado:

sudo cat /var/lib/mysql/server_audit.log

# 10. Averigua si en MongoDB se pueden auditar los accesos a una colección concreta. Demuestra su funcionamiento.

Si utilizamos MongoDB, es posible llevar un registro de los accesos realizados a una colección específica mediante la ejecución de un comando.

db.setLogLevel(level, component)

Utilizando este comando, podemos observar los accesos a la colección que se esté utilizando en tiempo real. Para llevar a cabo este proceso, es necesario crear una base de datos y una colección, y utilizar un nuevo usuario con permisos de administrador para realizar las operaciones correspondientes.

Al emplear esta técnica, se establece un nivel de detalle uniforme para todos los mensajes de registro.

En lo que respecta a su funcionamiento, esta técnica fija un único nivel de detalle para la generación de mensajes de registro. Si deseamos establecer diferentes niveles de detalle en una misma operación, será necesario emplear el comando "setParameter" para definir el parámetro "logComponentVerbosity". Es posible especificar la configuración de verbosidad en el archivo de configuración también.

Para realizar la tarea, se procederá a la ejecución del comando siguiente:

db.setLogLevel(3, "accessControl")

En términos prácticos, esto implica que se establecerá un nivel de detalle o verbosidad de 3 para el componente "accessControl". Es decir, que se visualizarán los mensajes correspondientes a errores, advertencias e información.

A continuación, se procederá a iniciar sesión con el usuario "admin" previamente creado y, posteriormente, se creará un nuevo usuario con permisos de administrador.

# Accedemos a la consola de mongo
mongosh
# Accedemos al administrador
use admin
# Creamos un nuevo usuario administrador
db.createUser ({user: 'olucas', pwd: 'root', roles: [{role: 'root', db: 'admin'}]})

En este momento, se realizará la conexión utilizando el nuevo usuario que se ha creado.

mongosh -u olucas -p root --authenticationDatabase admin

Seguidamente, se crearán dos colecciones denominadas "1curso" y "2curso", y se procederá a insertar las siguientes asignaturas en cada una de ellas:

use audit

db.createCollection("Primero")
db.createCollection("Segundo")

db.Primero.insertMany([{nombre: "Sistemas Operativos", horas: "6"}, {nombre: "Lenguajes de Marcas", horas: "4"}, {nombre: "Redes", horas: "5"}, {nombre: "Bases de Datos", horas: "6"}, {nombre: "Formación y Orientación Laboral", horas: "4"}, {nombre: "Fundamento de hardware", horas: "4"}])
db.Segundo.insertMany([{nombre: "Implantación de Aplicaciones Web", horas: "6"}, {nombre: "Cloud Computing", horas: "4"}, {nombre: "Seguridad", horas: "3"}, {nombre: "Empresas", horas: "4"}, {nombre: "Administración de Sistemas Operativos", horas: "6"}, {nombre: "Administración de Sistemas Gestores de Bases de Datos", horas: "3"}])

Para visualizar los accesos realizados a la colección que se esté utilizando en ese momento, se ejecutará el siguiente comando: "db.setLogLevel(3, "accessControl")".

Para verificar que la técnica se encuentra funcionando adecuadamente, se han realizado dos intentos de inicio de sesión, uno de forma correcta y otro de forma incorrecta.

Para confirmar que la auditoría se está llevando a cabo de forma correcta, se revisarán los registros de MongoDB desde la consola utilizando un formato JSON para su mejor legibilidad. Antes de proceder con esta tarea, se deberá haber instalado la herramienta jq.

apt install jq

cat /var/log/mongodb/mongod.log | jq

De este modo, se puede comprobar que se han registrado tanto el acceso correcto como el incorrecto en los registros de auditoría.
